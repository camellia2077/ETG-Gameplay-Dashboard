# Items 页面性能优化

本文只记录 `General -> Items` 页面及其相关长列表在本轮完成的性能优化。其他页面或运行时模块的性能优化应单独记录，避免混淆不同优化的收益和风险。

## 优化范围

本轮关注两个主要阶段：

1. 打开 Items 页面时的物品目录准备。
2. Items 页面浏览、筛选和滚动时的列表绘制。

## 优化前现象

> [!NOTE]
> 以下性能测试数据与基线日志均在 **Debug 构建版本** 下测试采集得出（Debug 编译包含额外的校验与符号支持）， Release 构建版本下的实际性能表现会进一步优于此数据。

性能日志显示，Items 页面包含约 670 个物品。打开页面后：

- 物品目录首次构建约消耗 289ms。
- 首次列表绘制约消耗 210ms。
- 后续每帧绘制全部物品行，通常消耗约 20–40ms。
- 这会造成打开控制面板时的明显卡顿和持续低帧率。

相关基线日志通过 `EnablePerformanceVerboseLogs` 获取，详见 [Performance Logging](./logging-performance.md)。

## 已完成的优化

### 1. 长列表虚拟化

Items 列表现在只绘制当前滚动视口附近的物品行，并保留少量上下缓冲行。列表的完整高度仍用于滚动条计算，但不可见行不会创建 GUI 控件、绘制图标或绘制文本。

### 2. 筛选结果缓存

Items 的筛选结果现在按搜索文本和筛选条件缓存。只有搜索文本、类别、品质、枪械类别、被动子类别或主动冷却类型发生变化时，才重新生成筛选数组。

普通滚动帧直接复用缓存，避免重复遍历目录和创建临时数组。

### 3. 物品目录缓存

物品目录按当前 UI 语言缓存。目录提供器不再在每次调用时重新扫描并解析所有游戏物品；同一语言下后续页面打开、Loadout 操作和 Items 刷新会复用目录结果。

如果 UI 语言发生变化，目录会重新生成以更新本地化显示名称。

### 4. 单行绘制和图标绘制

Items 物品行做了以下调整：

- 可见行之外不再绘制背景、按钮、文本和图标。
- 成功解析图标时不再额外绘制重复的图标背景框。
- 图标使用缓存的游戏精灵数据。
- 图标在约 `64×40` 的区域内保持原始宽高比并居中显示，避免横向物品被压缩成正方形。

## 交互和行为约束

- 搜索、类别筛选、品质筛选和滚动行为保持不变。
- 鼠标点击和手柄焦点导航保持不变。
- Grant、Start Items 和 Random Pool 模式的操作保持不变。
- 目录内容仍来自游戏运行时数据，没有新增持久化格式。

## 后续优化边界

本记录不包含以下内容：

- 其他运行时服务的 Update 性能。
- 附近物品提示、角色切换或房间功能的性能。
- Items 目录首次生成本身的进一步分帧或后台化处理。

后续如果处理这些内容，应分别建立独立的性能记录并提供各自的基线和验证结果。
